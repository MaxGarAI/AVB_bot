# AVB / Dealer.getbiz.me MVP Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Собрать MVP AI BizDev-агента `dealer.getbiz.me`, который проводит интервью с партнером, извлекает структурированные данные, считает score, генерирует summary и сохраняет все в SQLite.

**Architecture:** Для MVP делаем один backend-сервис на Python с HTTP API, SQLite как основную БД, OpenRouter как LLM-провайдера и простую web-страницу чата. Логику разделяем на модули: session/token management, conversation state machine, extraction, scoring, summary, persistence.

**Tech Stack:** Python 3.11, FastAPI, Uvicorn, SQLAlchemy 2.x, Alembic, Pydantic v2, Jinja2, HTMX или vanilla JS, SQLite, OpenRouter API, pytest.

**LLM Provider:** OpenRouter
**LLM Model:** `meta-llama/llama-3.1-8b-instruct:free`
**Database:** SQLite (локальный файл `app.db`)
**Virtual Environment:** `.venv` в корне проекта

## 3. Архитектура MVP

### 3.1 Основные runtime-компоненты
1. **Conversation API** — принимает сообщения пользователя и возвращает следующий вопрос.
2. **Session Service** — создает токен, хранит контекст sales, статус и stage.
3. **Extraction Service** — после каждого user message обновляет структуру профиля.
4. **State Machine** — решает, на какой стадии находится интервью и можно ли двигаться дальше.
5. **Scoring Service** — считает `partner_score` по собранным данным.
6. **Summary Service** — создает executive/sales summary.
7. **Admin Read API** — отдает результаты интервью.

### 3.2 Поток выполнения
1. Sales создает interview session.
2. Система генерирует `session_token`.
3. Partner открывает `/chat/{token}`.
4. Backend собирает system prompt = базовая персона Dealer + custom instructions.
5. Пользователь отправляет сообщение.
6. Conversation Agent формирует следующий вопрос.
7. Extraction Agent извлекает JSON из полной истории или окна истории.
8. State machine обновляет stage и completion score.
9. Когда данных достаточно — генерируются score и summaries.
10. Интервью переводится в `completed`.

## 5. Stage machine для MVP

Использовать фиксированный enum:
- `INIT`
- `BUSINESS_DISCOVERY`
- `AUDIENCE_DISCOVERY`
- `SEGMENT_DISCOVERY`
- `NEEDS_DISCOVERY`
- `CHANNEL_DISCOVERY`
- `PARTNERSHIP_DISCOVERY`
- `COMPLIANCE_DISCOVERY`
- `SUMMARY`
- `COMMIT`
- `FINISHED`

### Правила переходов
- Из `BUSINESS_DISCOVERY` выходим, когда заполнены минимум: `business_name`, `industry`, `city/state` или `website`.
- Из `AUDIENCE_DISCOVERY` выходим, когда есть минимум 2 из 3: `monthly_customers`, `annual_customers`, `repeat_rate`.
- Из `SEGMENT_DISCOVERY` выходим, когда есть минимум 1 качественный сегмент.
- Из `NEEDS_DISCOVERY` выходим, когда есть минимум 3 customer insights.
- Из `CHANNEL_DISCOVERY` выходим, когда есть минимум 1 подтвержденный канал.
- Из `PARTNERSHIP_DISCOVERY` выходим, когда определена готовность к хотя бы одному типу дистрибуции.
- Из `COMPLIANCE_DISCOVERY` выходим, когда определены consent/restrictions или явно зафиксировано `unknown`.
- `SUMMARY` запускается только если `completion_score >= 70`.
- `COMMIT` переводит сессию в `FINISHED`.

## 6. OpenRouter интеграция

### env-переменные
Ожидаемые переменные:
- `OPENROUTER_API_KEY`
- `OPENROUTER_MODEL=meta-llama/llama-3.1-8b-instruct:free`
- `APP_ENV=dev`
- `DATABASE_URL=sqlite:///./app.db`

### HTTP-конфиг
- Base URL: `https://openrouter.ai/api/v1/chat/completions`
- Request timeout: 60 sec
- Retry: 2 попытки только на network/429/5xx
- Temperature: 0.2–0.4
- JSON-экстракцию делать либо через `response_format`, либо через жесткий prompt + `json.loads` validation

### Важный компромисс MVP
Бесплатная `llama-3.1-8b-instruct:free` может быть менее стабильной в строгом JSON output. Поэтому extraction и summary надо делать в отдельных вызовах с очень строгими схемами, а не пытаться получить всё за один запрос.

## 7. REST endpoints MVP

### Session creation
- `POST /api/sessions`
  - body: `sales_manager_name`, `source_campaign`, `custom_prompt_instructions`
  - response: `session_token`, `chat_url`

### Chat UI
- `GET /chat/{token}`
  - отрисовать страницу чата

### Send message
- `POST /api/chat/{token}/messages`
  - body: `message`
  - response:
    - assistant message
    - current stage
    - completion score
    - partial structured data

### Session details
- `GET /api/sessions/{token}`
  - transcript + extracted data + score + summaries

### Admin list
- `GET /api/admin/sessions`
  - список с фильтрацией по status/stage

### Health
- `GET /health`

## 9. Формула completion score для MVP

Предлагаемая формула:
- Business Profile completeness — 20%
- Audience Profile completeness — 20%
- Segment quality — 20%
- Needs insights — 15%
- Channels — 15%
- Partnership/Compliance — 10%

Порог завершения:
- `>= 70` -> можно переходить к summary
- `>= 85` -> high-quality completed interview

---

## 10. Формула partner score для MVP

Итог `0..10`:
- Audience Size — 0..2
- Audience Trust — 0..2
- Reachability — 0..2
- Cross-Sell Potential — 0..2
- Data Quality — 0..2

### Простая логика
- Большая и регулярная аудитория -> больше баллов.
- Есть email/SMS/CRM/social reach -> больше баллов.
- Есть четкие сегменты и боли -> больше баллов.
- Есть willingness to recommend -> больше баллов.
- Много unknown/estimated fields -> меньше баллов.

## 12. Безопасность и ограничения MVP

- Секреты только через `.env`.
- Не логировать API ключ и полный сырой headers dump.
- Токен сессии должен быть случайным и трудноугадываемым.
- Никаких eval/unsafe JSON parsing.
- Для SQLite обязательно делать резервную копию файла `app.db`.

### Backup стратегия для SQLite
- dev: ручной backup `app.db` перед крупными миграциями
- MVP deploy: ежедневная копия `app.db` в `backups/`
- При изменении схемы — backup before migration

## 16. Definition of Done для MVP

MVP готов, если:
- создается session token;
- открывается chat page;
- бот задает по одному вопросу;
- transcript сохраняется;
- extracted JSON сохраняется;
- есть хотя бы один audience segment;
- есть хотя бы один channel;
- считается partner score;
- генерируются executive и sales summary;
- все данные можно посмотреть через admin/session endpoint;
- happy-path тесты проходят.
